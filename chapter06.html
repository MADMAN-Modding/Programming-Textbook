Chapter 6

Objects

ES6 classes: constructor, methods, this, and instances.

Alright, alright, here we go. Up to now we've kept data in loose variables — x, y, xspeed, yspeed — and behavior in separate functions. That works for one bouncing ball. But what if we want fifty bouncing balls? Fifty x's? Fifty y's? Yikes. We need a way to bundle data and behavior together. That's an object. And in modern JavaScript, we make objects with classes.

The big idea #

An object is a thing that has properties (data) and methods (functions that belong to it). A class is the blueprint that describes how to make those things.

Think of it like a cookie cutter. The cutter (class) describes the shape. Every time you press it into dough, you get a new cookie (instance). They all share the same shape, but each cookie is its own cookie.

Coding Train 6.1 — Intro to Object-Oriented Programming

Defining a class #

Here's a Ball class. Read the whole thing first, then we'll break it down.

class Ball {
  constructor(x, y, diameter) {
    this.x = x;
    this.y = y;
    this.diameter = diameter;
    this.xspeed = random(-3, 3);
    this.yspeed = random(-3, 3);
  }

  move() {
    this.x += this.xspeed;
    this.y += this.yspeed;
  }

  bounce() {
    if (this.x < 0 || this.x > width)  this.xspeed *= -1;
    if (this.y < 0 || this.y > height) this.yspeed *= -1;
  }

  show() {
    fill(237, 34, 93);
    ellipse(this.x, this.y, this.diameter, this.diameter);
  }
}

Three things to notice:

  1. The class keyword, followed by the class name (capitalized — that's the convention).
  2. A special function called constructor(). It runs once, when you create a new ball, and it sets up the initial state.
  3. Other functions (move, bounce, show) called methods. They are this class's behaviors.
Why no commas? Inside a class, methods are not separated by commas. Just write them one after the other. This trips up beginners coming from older JavaScript object syntax — leave the commas out!

What is "this"? #

The keyword this means "the specific object that this method is being called on." When you write this.x = x in the constructor, you're saying "save the incoming x value as a property of this specific ball."

Without this, your class wouldn't know whether you mean ball #1's x or ball #2's x. The this makes it crystal clear: this method, right now, is operating on this specific instance.

Forgot this.? The single most common bug in OOP code. If you write x += xspeed inside a method instead of this.x += this.xspeed, JavaScript will look for global x and xspeed variables. Spoiler: they don't exist. Your ball won't move and your console will fill with errors.

Creating instances with new #

Defining a class doesn't create anything yet — it's just a blueprint. To actually make a ball, you use the new keyword:

let ball;

function setup() {
  createCanvas(400, 400);
  ball = new Ball(200, 200, 40);
}

function draw() {
  background(220);
  ball.move();
  ball.bounce();
  ball.show();
}

The variable ball now holds an instance of the Ball class. You access its methods with dot notation: ball.move(), ball.show(). You can also peek at its data: ball.x, ball.diameter.

Coding Train 6.2 — Defining a Class

Multiple instances — the payoff #

Here's where it gets really good. Now that you have a class, making fifty balls is just fifty calls to new Ball(...). Each one has its own x, y, and speed. They all know how to move and draw themselves.

let ball1, ball2, ball3;

function setup() {
  createCanvas(400, 400);
  ball1 = new Ball(100, 100, 30);
  ball2 = new Ball(200, 200, 50);
  ball3 = new Ball(300, 100, 20);
}

function draw() {
  background(220);
  ball1.move();  ball1.bounce();  ball1.show();
  ball2.move();  ball2.bounce();  ball2.show();
  ball3.move();  ball3.bounce();  ball3.show();
}

Three balls, each with their own state, all running through the same logic. Pretty good, right? But you can probably feel that this is going to get out of hand fast. ball50? ball100? In Chapter 7 we'll fix this with arrays, and you'll be making thousands.

Coding Train 6.3 — Objects and Instances

Why classes and not the old way? #

You might see older Dan Shiffman videos that use "constructor functions" — a different way of making objects that JavaScript had before ES6 added classes in 2015. Both work. But classes are clearer, modern, and what you'll see in every other language (Java, Python, C++, Swift, …). It's also what's on the AP CSA exam. We're going with classes.

A naming convention worth keeping Class names are PascalCase (Ball, Particle, FlappyBird). Instance variables are camelCase (ball, myBird). Methods are camelCase too. This isn't enforced by JavaScript — it's just how people read code.

Common Beginner Mistakes #

Forgetting this. The #1 mistake. Inside a method, every reference to the object's own properties needs this. in front. Not "x", not "xspeed" — this.x and this.xspeed.
Forgetting new let ball = Ball(100, 100, 40); — without new, you've just called the class as a function, and you'll get a confusing error. Always use new ClassName(...).
Putting commas between methods Inside a class { ... } block, methods are not separated by commas. Just stack them one after another.
Defining the class inside setup() Define your classes at the top level of your sketch, outside of setup() and draw(). Inside setup() you should be using them with new, not defining them.
Lowercasing the class name JavaScript won't force you to capitalize class names, but everybody does. new ball(...) looks like a function call; new Ball(...) screams "this is a class."
